# 要件定義書 (anaslo-scraper-ja)

本要件定義書は、パチスロデータ分析サイト「アナスロ」から愛媛県松山市の店舗データを自動収集し、SQLiteデータベースに蓄積してStreamlitダッシュボードで分析するためのシステム要件を定義します。

---

## 1. システム概要

本システムは、ユーザーが指定した店舗の出玉実績データをクローリングし、データベースに保存するとともに、集計・分析結果をグラフや表で確認できるローカルWebアプリケーションです。

---

## 2. 用語定義

本システムで使用する専門用語および計算ルールの定義は以下の通りとします。

| 用語 | 定義 / 算出式 |
| :--- | :--- |
| **日付の末尾** | 営業日（日：1〜31日）の1の位の数字（0〜9）。 |
| **7のつく日** | 毎月 7日、17日、27日。 |
| **ゾロ目の日** | 毎月 11日、22日。 |
| **勝率** | 差枚数がプラス（`difference > 0`）の台数 ÷ 総台数。 |
| **合成確率** | 対象サイトに記載されている合成確率の文字列（例: `1/140.2`）。BB確率、RB確率も同様。 |
| **バックフィル** | 過去の特定期間（例: 過去30日分）の営業データを遡って一括取得すること。 |
| **リフレッシュ** | すでに登録済みの営業日データを、最新のサイト情報で上書き更新すること。 |

---

## 3. 動作環境・技術要件

*   **OS**: Windows 10/11 (日本語環境)
*   **実行環境**: Python 3.10以上
*   **データベース**: SQLite 3 (標準ライブラリ使用)
*   **可視化フレームワーク**: Streamlit
*   **クローリングライブラリ**: `requests` および `BeautifulSoup4` (ボット制限検知時はエラー終了し、ログを出力する)

---

## 4. 機能要件 (EARSフォーマット)

### 4.1 データ収集 (クローラー)
*   **[REQ-COL-01] 店舗設定**
    *   システムは、設定ファイル（`config.json`）に記述された店舗名とURL（データ一覧ページのURL）に基づいてデータを収集すること。
*   **[REQ-COL-02] 自動日付取得**
    *   システムは実行時、対象店舗のデータ一覧ページから「まだデータベースに登録されていない日付」を自動で判別し、その日付の個別データページをクローリングすること。
*   **[REQ-COL-03] 二重登録の防止**
    *   システムは、すでにデータベースに登録済みの営業日のデータは、明示的な上書き指定がない限り収集をスキップすること。
*   **[REQ-COL-04] アクセス負荷軽減**
    *   クローラーは、ページ遷移時やリクエストの間に必ず3〜5秒のランダムなスリープを挟み、対象サイトに負荷をかけないこと。
*   **[REQ-COL-05] 過去データ一括取得（バックフィル）**
    *   システムは、ユーザーが指定した過去の営業日数（例: 直近30日分）を遡って一括で収集・保存できること。
*   **[REQ-COL-06] データの上書き更新（リフレッシュ）**
    *   システムは、特定の日付を明示的に指定された場合、すでにデータベースに存在するデータを一度削除し、サイトから最新データを再取得して上書き保存できること。
*   **[REQ-COL-07] パチスロデータ限定**
    *   システムは、収集対象をスロット（パチスロ）データのみに限定し、パチンコデータは収集・保存の対象外とすること。
*   **[REQ-COL-08] 複数店舗の直列実行**
    *   システムは、config.jsonに複数店舗が定義されている場合、1店舗ずつ順番に直列で処理を実行すること。また、1つの店舗でクローリングエラーが発生しても、他の店舗の処理は中断せず最後まで進めること。

### 4.2 データベース (SQLite)
*   **[REQ-DB-01] データの永続化**
    *   システムは、収集したデータを `anaslo_data.db` というSQLiteファイルに保存すること。
*   **[REQ-DB-02] 保存データ項目**
    *   データベースには、以下のデータを正規化して保存すること。
        *   **日別全体サマリー**: 日付、曜日、総差枚、平均差枚、平均ゲーム数、勝率、勝ち台数（差枚>0の台数）、総台数
        *   **台別詳細**: 台番号、機種名、ゲーム数、差枚数、BB回数、RB回数、合成確率、BB確率、RB確率、台番号の末尾1桁（自動計算）

### 4.3 分析・可視化 (Streamlitダッシュボード)
*   **[REQ-ANA-01] 機種別実績分析**
    *   ユーザーが特定の機種を選択した際、その機種の「曜日ごとの平均差枚・勝率」および「日付の末尾ごとの平均差枚・勝率」をグラフと表で可視化すること。
*   **[REQ-ANA-02] 特定日付・イベント日の分析**
    *   ダッシュボードは、店舗全体または特定機種の「7のつく日」「ゾロ目の日」などのイベント日の傾向を、棒グラフを用いて通常日（イベント日以外）と比較できること。
*   **[REQ-ANA-03] 台番号末尾の分析**
    *   ダッシュボードは、台番号の末尾（0〜9）ごとの平均差枚および勝率を算出し、特定の末尾に設定投入傾向があるか確認できるグラフを表示すること。
*   **[REQ-ANA-04] 強機種ランキング**
    *   ダッシュボードは、ユーザーが指定した期間（直近30日、60日、90日、またはカスタム日付範囲）で平均差枚・勝率が高い「強い機種」をランキング形式で表示すること。期間のデフォルト値は30日とすること。

## 5. 非機能要件

### 5.1 パフォーマンス
*   **[REQ-NFR-PERF-01] クロール処理速度**
    *   1日分のデータパース処理（約200〜400台分）は、通信時間を除き3秒以内で完了すること。
*   **[REQ-NFR-PERF-02] 画面応答時間**
    *   ダッシュボードで店舗や機種を切り替えた際、1000日分のデータが存在する場合でも、集計・グラフ描画が3秒以内に完了すること。

### 5.2 セキュリティ・マナー
*   **[REQ-NFR-SEC-01] User-Agent設定**
    *   クローラーからのアクセス時は、一般的なWebブラウザ（Chrome等）のUser-Agentを設定し、ボットとみなされブロックされるのを防ぐこと。
*   **[REQ-NFR-SEC-02] 収集ルール遵守**
    *   クローリングのアクセス間隔を厳守し、サイトに急激な負荷を与えないこと。

### 5.3 データ容量・保持方針
*   **[REQ-NFR-DAT-01] データ保持期間**
    *   収集したデータは、ディスク容量が許す限り永続的に保持すること（自動削除は行わない）。
*   **[REQ-NFR-DAT-02] バックアップ**
    *   システムの起動時（クローラーまたはダッシュボード起動時）に、データベースファイル `anaslo_data.db` のコピーを `backup/` ディレクトリに自動生成すること（直近5世代をローテーション保持）。

### 5.4 可用性・エラーハンドリング
*   **[REQ-NFR-ERR-01] 通信エラー時の自動リトライ**
    *   HTTPリクエストがタイムアウト（15秒）またはネットワークエラーになった場合、最大3回、10秒間隔で自動リトライを行うこと。
*   **[REQ-NFR-ERR-02] サイト構造変化の検知**
    *   HTML構造の変化等によりデータのパースに失敗した場合、クラッシュせずに「パース失敗エラー」としてログに記録し、プログラムを安全に終了すること。

### 5.5 ログ
*   **[REQ-NFR-LOG-01] ログファイル**
    *   実行ログを `logs/anaslo.log` に出力すること。エラー発生時はスタックトレースを含めて記録し、ログファイルが10MBを超えた場合はローテーションすること。

### 5.6 ライブラリバージョン
*   **[REQ-NFR-VER-01] 依存バージョン**
    *   主要依存ライブラリは以下の範囲で動作確認を行うこと。
        *   `requests` >= 2.31.0
        *   `beautifulsoup4` >= 4.12.0
        *   `streamlit` >= 1.30.0
        *   `pandas` >= 2.0.0

## 6. 制約事項・リスク

*   **[REQ-CON-01] 法的・利用規約上のリスク**
    *   本システムは、個人による出玉データ研究・分析目的でのみ利用すること。取得したデータを二次配布または商用利用しないこと。
    *   対象サイト（アナスロ）におけるコピー禁止・右クリック禁止などのスタンスを尊重し、アクセス頻度を極小化（1店舗あたり1日1アクセス）して動作させること。
*   **[REQ-CON-02] SQLiteの同時アクセス制限**
    *   クローラーの書き込み処理と、Streamlitの読み込み処理が競合しないよう、`sqlite3` 接続時は適切なタイムアウト設定（`timeout=20.0`）を行い、データベースロック（`database is locked`）エラーを回避すること。

## 7. 受け入れ基準 (Acceptance Criteria)

### AC-1: クローラーの動作（正常系・追加機能）
*   `python run.py --collect` を実行した際、`config.json` に設定された店舗（大盛空港通り店）の未登録の日付のデータが自動で取得され、`anaslo_data.db` に正しく保存されること。
*   すでに登録済みの日のデータは「登録済み」としてスキップされること。
*   バックフィルコマンド（例: `python run.py --backfill 5`）を実行した際、指定された過去5日分のデータがエラーなく一括収集されること。
*   上書き更新コマンド（例: `python run.py --refresh YYYY-MM-DD`）を実行した際、該当日のデータが一度クリアされ、再取得されたデータに置き換わること。
*   複数店舗が設定されているとき、1店舗目が通信エラー（モックテスト）で失敗しても、2店舗目の収集処理が正しく実行されること。

### AC-2: データ整合性
*   `unit_details` テーブルに保存された各台のデータにおいて、台番号が `327` のときは `unit_number_tail` が `7` として正しく登録されていること。
*   `daily_summaries` テーブルの `win_units` に、その日の差枚数がプラス（`difference > 0`）だった台の合計数が正しく保存されていること。
*   機種名に「♪」や「★」といった特殊文字を含む機種（例: `スマスロ北斗の拳` 等）が文字化けせずDBに格納されていること。

### AC-3: ダッシュボードの動作（正常系・追加機能）
*   `streamlit run app.py`（または起動スクリプト）を実行した際、ブラウザにダッシュボードが表示されること。
*   期間選択UIにて、デフォルトで直近30日のデータが対象となり、セレクトボックスで「60日」「90日」や「カスタム範囲」へ正しく切り替えられること。
*   機種選択時、「曜日別」「日付の末尾別」「台番号の末尾別」のグラフがすべてエラーなく描画され、勝率が「（差枚数プラス台数÷総台数）」の計算式通りに％表示されていること。
*   「7のつく日」および「ゾロ目の日」を選択した際、通常営業日との平均差枚数の比較グラフが正しく描画されること。

### AC-4: 異常系ハンドリング
*   ネットワークを切断した状態でクローラーを実行した際、10秒間隔で3回リトライが走り、最終的に「接続エラー」としてログに出力されて安全に終了すること。
*   `config.json` が破損している、または店舗URLが不正な場合、エラーが画面に表示され、クローラーが起動しないこと。
*   アナスロのサイトがメンテナンス画面（HTTP 503等）のとき、ログにエラーを記録し、DBを汚さずに終了すること。
